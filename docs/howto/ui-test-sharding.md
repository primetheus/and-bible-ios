# UI Test Sharding

This guide explains how the repository currently splits `AndBibleUITests`
across CI jobs, how to validate the same shape locally, and which assumptions
are worth preserving unless you have real runtime data in hand.

Use this together with [building-and-testing.md](building-and-testing.md) when
you are debugging UI failures or considering CI runtime changes.

## Current CI Model

The iOS workflow currently runs UI tests through a planned matrix generated by:

- `scripts/build_ui_test_shards.py`
- `.github/workflows/ios-ci.yml`

Current workflow inputs:

- `UI_TEST_MIN_SHARD_COUNT = 2`
- `UI_TEST_TARGET_SHARD_DURATION_SECONDS = 600`
- `UI_TEST_TIMEOUT_MINUTES = 90`
- `UI_TEST_TIMINGS_FILE = scripts/ui_test_timings.json`

Important nuance:

- the planner is capable of dynamic shard counts
- in practice, with the current manifest and suite size, it usually behaves as
  a balanced 2-way split

It is better to describe the current system as "planner-capable but usually a
balanced 2-way split" unless the timing manifest is being refreshed from real
runs and the shard count is actually moving because of current evidence.

## How CI Actually Executes

### Workflow shape

1. discover test identifiers from `AndBibleUITests/AndBibleUITests.swift`
2. load estimated durations from `scripts/ui_test_timings.json`
3. build a shard matrix via `scripts/build_ui_test_shards.py`
4. for each shard:
   - run `build-for-testing`
   - build the host-side UI fixture tool
   - run one `test-without-building` invocation using the shard's
     `-only-testing:` selection list

This means each UI shard is one `xcodebuild test-without-building` run, not a
loop of smaller groups.

That's an important distinction because of the following:

- fail-fast behavior is different
- runtime attribution is clearer
- "all tests in a shard executed" is only true because the shard is a single
  `xcodebuild` invocation

## Local Validation Guidance

If you want an honest read on CI, a few targeted tests are usually not enough.

Preferred order:

1. `build-for-testing`
2. non-UI `test-without-building` with `-skip-testing:AndBibleUITests`
3. shard-shaped UI validation using the same generated `-only-testing:` list
4. optionally a full serial `-only-testing:AndBibleUITests` run when you need
   maximum confidence

Why this matters:

- a full serial local UI run covers the same 39 UI tests, but its wall-clock
  runtime is not comparable to CI because CI runs shards in parallel
- a targeted subset can prove a specific fix, but it does not prove the shard
  or full-suite shape is clean

## Runtime Interpretation

Do not compare these numbers as if they were the same thing:

- full local serial UI runtime
- one CI shard runtime
- whole CI workflow wall-clock

The full serial local run is useful for:

- end-to-end confidence
- proving the suite is not hanging
- catching cross-test contamination

It is not the right measure for:

- whether a 2-shard split is well balanced
- whether a 90-minute per-shard timeout is justified

For sharding decisions, use shard-shaped runs and current CI data.

## Guardrails

1. Try not to change shard count or timeout based only on intuition.

   Any change to `UI_TEST_MIN_SHARD_COUNT`,
   `UI_TEST_TARGET_SHARD_DURATION_SECONDS`, or
   `UI_TEST_TIMEOUT_MINUTES` should be backed by current CI/runtime evidence.

2. Be careful about treating the current timing manifest as authoritative unless
   it has been refreshed recently.

   `scripts/ui_test_timings.json` can go stale. When it is stale, the planner
   still works as a rough balancer, but the "target shard duration" becomes
   aspirational rather than predictive.

3. Avoid replacing the single-invocation shard model with ad hoc per-test loops
   unless you are also updating the docs and the reasoning behind the change.

   That would materially change failure behavior and runtime interpretation.

4. If you touch shard planning or test-selection scripts, update:

   - this file
   - `docs/howto/building-and-testing.md` when developer workflow changes
   - `UI-TEST-ANALYSIS.md` if the operational conclusions materially change

## What This Guide Does Not Claim

This guide does not claim that:

- the current 90-minute timeout is optimal
- the current timing manifest is fresh
- the current 2-shard split is the final long-term design

It only documents the system as it exists now, so future changes can be judged
against an explicit contract instead of fuzzy repo memory.
